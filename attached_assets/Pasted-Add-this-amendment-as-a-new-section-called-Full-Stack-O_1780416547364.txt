Add this amendment as a new section called **Full-Stack Observability and Layer Visibility for UI Inspection**. The purpose of this section is to give operators, developers, and admins direct visibility into every meaningful layer of the system in the product UI: user request, run, agent, context, policy, model call, tool call, adapter, memory write, approval, artifact, event, and evaluation. In agent systems, traces become the primary source of truth because failures often come from reasoning, routing, or context quality rather than a traditional code crash, and modern observability guidance consistently emphasizes traces, metrics, and logs across the full execution tree. ## X+2. Full-Stack Observability and Layer Visibility for UI Inspection

### X+2.1 Purpose

The system must provide **UI-level visibility into all layers** so that users can inspect what happened, why it happened, what context was used, which model or agent made which decision, which tools were called, what policies applied, and where time, tokens, or failures accumulated. Observability for LLM and agent systems requires preserving the full execution context of each request, including prompts, responses, tool calls, and their relationships, rather than relying on generic logs alone. [opentelemetry](https://opentelemetry.io/blog/2024/llm-observability/)

This observability layer must support:
- debugging,
- operations monitoring,
- quality review,
- cost analysis,
- compliance auditing,
- safety review,
- dataset creation for evals,
- UX visibility into run internals.

### X+2.2 Core principle

Every top-level user request or system trigger must map to **one root trace**, and everything else must hang beneath it as structured child observations. This trace-tree model is widely recommended because it lets teams inspect one request as a full execution graph spanning agents, model calls, retrieval, tools, and downstream actions. [blogs.oracle](https://blogs.oracle.com/ai-and-datascience/observability-for-multi-agent-systems)

The UI must expose visibility into these layers:
- request/thread layer,
- run layer,
- task graph layer,
- agent layer,
- model invocation layer,
- context assembly layer,
- memory layer,
- capability/tool/resource/prompt layer,
- adapter/network layer,
- approval/policy layer,
- artifact/output layer,
- eval/quality layer,
- audit/security layer.

### X+2.4 Trace model

The trace model must be hierarchical.

#### Root trace hierarchy

- Trace
  - Run observation
    - Task node observation
      - Agent run observation
        - Model call observation
        - Context assembly observation
        - Retrieval observation
        - Tool call observation
        - Policy check observation
        - Approval observation
        - Memory write observation
        - Artifact write observation
        - Eval observation

This tree must be reconstructable in the UI as an expandable timeline and as a graph. Agent observability guidance repeatedly recommends trace-centric debugging because the execution path itself is the system behavior. [youtube](https://www.youtube.com/watch?v=FDVdLrloFOw)

### X+2.5 Instrumentation requirements

Every service in the platform must emit traces, logs, and metrics, and the instrumentation model should align with OpenTelemetry conventions rather than inventing isolated proprietary telemetry formats. OpenTelemetry guidance for agent systems explicitly recommends standardized traces, metrics, and logs, plus compatibility with existing instrumentation libraries. [opentelemetry](https://opentelemetry.io/blog/2025/ai-agent-observability/)

#### Required instrumentation points

Emit an observation for:
- intent creation,
- policy bundle resolution,
- adapter discovery,
- context fragment retrieval,
- context pack assembly,
- agent start/end,
- model invocation,
- tool proposal,
- tool execution,
- resource read,
- prompt fetch,
- approval requested/resolved,
- memory write,
- compaction,
- artifact creation,
- eval execution,
- error handling,
- final response render.

#### Required metadata fields

Every observation must include:
- `tenant_id`
- `trace_id`
- `run_id` if applicable
- `agent_id` if applicable
- `agent_run_id` if applicable
- `node_id` if applicable
- `principal_id` if applicable
- `adapter_id` if applicable
- `capability_id` if applicable
- `model_name` if applicable
- `status`
- `started_at`
- `ended_at`
- `latency_ms`

### X+2.6 Snapshot requirements

The UI should support drill-down, so each observation type must store the right snapshots.

#### Model call snapshot

`input_snapshot_json` should include:
- system prompt or instruction reference,
- conversation messages,
- selected context summary,
- tool availability summary,
- response format schema ref.

`output_snapshot_json` should include:
- raw text,
- structured output,
- tool calls requested,
- usage and finish reason.

#### Context assembly snapshot

`input_snapshot_json` should include:
- intent,
- candidate fragment IDs,
- policy bundle ref,
- token budget.

`output_snapshot_json` should include:
- selected fragment IDs,
- rejected fragment IDs,
- scores,
- reserved tokens,
- resource handles.

#### Tool call snapshot

`input_snapshot_json` should include:
- tool name,
- arguments,
- policy context,
- approval state,
- delegated account ref.

`output_snapshot_json` should include:
- structured result,
- returned artifacts,
- error state,
- latency,
- retry count.

#### Policy check snapshot

Should include:
- matched rules,
- denied rules,
- redactions applied,
- approval requirement decision,
- risk score.

These detailed snapshots are essential because effective observability for agent systems depends on seeing the causal chain rather than only final outputs. [langfuse](https://langfuse.com/docs/observability/overview)

### X+2.11 Eval integration

Observability and evaluation must be connected. Production traces should be reusable as an eval dataset source, and evaluation records should attach back to traces and observations so teams can compare good vs bad trajectories over time. Trace-driven eval workflows are now a common best practice in agent development. [youtube](https://www.youtube.com/watch?v=reISMhbZ2XE)

Add capabilities to:
- mark traces as reference examples,
- label traces as success/failure,
- sample traces into offline eval sets,
- attach human review notes,
- compare two traces side by side,
- replay selected traces through newer prompts/models.

### X+2.13 Access control for observability

Observability must be deep but safe.

Rules:
- owners/admins can view all traces in tenant,
- members can view traces they initiated and traces they are granted access to,
- viewers get read-only filtered visibility,
- secret-bearing fields must be masked by default,
- raw prompts and tool payloads should be permission-gated,
- export settings are owner/admin only,
- audit views cannot be edited or deleted.

### X+2.14 Acceptance criteria

This amendment is complete only if:
- every run has a root trace,
- every major operation creates an observation,
- the UI can render a full execution tree,
- the UI can show context, model, tool, policy, and artifact layers separately,
- tokens, latency, and cost are visible per layer,
- eval records attach to traces and observations,
- users can filter and inspect traces by agent/model/tool/policy,
- telemetry can be exported to external observability systems,
- sensitive fields are masked according to access policy. [blogs.oracle](https://blogs.oracle.com/ai-and-datascience/observability-for-multi-agent-systems)

### X+2.15 Implementation order

Tell the coding platform to build this amendment in the following order:
1. `traces`
2. `observations`
3. instrumentation hooks in run/model/tool/context/policy paths
4. trace list page
5. trace detail page with execution tree
6. metrics aggregation
7. specialized dashboards
8. eval linkage
9. export configuration
10. masking and access control

This order works because trace capture is the foundation; metrics, dashboards, and evals only become reliable once the system records complete, structured traces first. [langfuse](https://langfuse.com/docs/observability/overview)
